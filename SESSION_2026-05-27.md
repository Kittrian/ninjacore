## Two data-loss events during testing (recovered)

Both hit client 17079358 (Kenneth West) during failed bridge attempts:

1. **v1 bridge** — passed `accounts: []` / `progress: []` to `UpdateClientService` which propagated through `mergeReportJsonl` → wiped 618 KB `_report.jsonl` to ~41 KB. Recovered via user-initiated Flow 1 Get Report refresh.
2. **v2 bridge** — called shared `writeReportDataEntry` helper which either race-conditioned or got an empty `readReportDataJsonl()` result → wiped 1.26 MB `_rd.jsonl` to 183 bytes. Recovered via user-initiated Flow 1 refresh.

Lesson captured in the architecture pivot: **don't bridge data between flows by copying through shared file-mutation helpers**. Instead, add reader-side adapters that union/fall through both storage shapes. This is what the `*Resolver` files do, and the architecture is now stable.

The bridge code was fully torn out from both sides:
- ninjacore-web `server.mjs` reverted to pre-bridge backup
- Express `cas.ts`, `InternalReportSyncController.ts`, `internalRoutes.ts` moved to `/tmp/bridge-removed/` on Contabo

## Speed / cache work — what shipped vs. what didn't

| Layer | Status | Notes |
|---|---|---|
| Redis `fastQuery()` cache | ✅ Live | `clients:show:{cid}:{all}` keys, 60 s TTL, hit ≈ 1 ms |
| `singleFlight()` coalescing | ✅ Live | Used inside `fastQuery()`; prevents stampede |
| `timed()` metrics | ✅ Live | Logs cold reads >50 ms to journal |
| Existing cache invalidation hook | ✅ Verified | `invalidateCache('clients:*')` in `ClientController.update/store/remove` covers the new keys (Redis glob match) |
| HTTP keepalive agents | ⚠️ **Built, not wired** | `getKeepaliveAgents()` exists in `surrealFast.ts` but Node 18 undici fetch doesn't use them unless `surql()` is rewritten to pass `dispatcher`. ~30 ms per cold read still recoverable by finishing this. |
| Tokio | ❌ Not added | Tokio is Rust async runtime; ninjacore (Rust/Axum) already uses it. No applicable Tokio addition to Express (Node). |
| Cloudflare Workers / KV / Durable Objects | ❌ Not added | Would add ~50 ms vs Redis loopback. R2 already used directly. |

## Open items

1. **Finish HTTP keepalive wiring** — rewrite Express `surql()` in `lib/surreal.ts` to pass `dispatcher` from `surrealFast.ts` agents. Saves ~30 ms per cold lookup.
2. **3-doc upload test** — blocked: Chrome MCP `file_upload` only accepts user-shared files; my generated `/tmp/test-doc-*.png` get rejected and the user's inline-pasted PNGs aren't accessible as file paths. Need user to save the 3 blank docs to disk and share the paths.
3. **ninjacore-web doesn't persist address fields** — user typed "100 Test Avenue / Austin / TX / 78701" into form; saved row on Hetzner Surreal shows `addresses: None`, `current_address: None`. Saver in `server.mjs` is dropping the address block.
4. **WebP vs PNG vs binary discrepancy** — code comment says "PNG lossless" but `processAndUploadDocs` actually does `sharp().webp({quality:82})` (lossy). `clients.client_bin` has 3,355 populated rows of 192-byte crypto envelopes (not image bytes) and no live writer. Worth a focused fix if forensic quality of ID docs matters for letter generation.
5. **Vue SPA source isn't on any of the 3 servers** — searched ALL of frontend Contabo, backend Contabo, Hetzner, plus GitHub. Memory was right: source is on previous developer's machine or a never-cloned private repo. All current fixes are server-side only because of this.
6. **Schema widening for ninjacore-style string client_ids on int-only tables** (`extra_infos`, `report_data_entries`) — blocked by auto-mode classifier (correctly: it's a global type change). Resolvers work around it via type-safe fallbacks, but if you ever want ninjacore clients to have native rows in those tables, the schema needs to be widened.
